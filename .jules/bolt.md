## 2024-05-19 - Fast Path URL Hostname Extraction
**Learning:** When attempting to avoid `new URL()` instantiation overhead by manually parsing standard `http://` and `https://` schemas, simply scanning for the first `/` is fundamentally flawed and breaks URL specifications. The "authority" (hostname) boundary can also be terminated by a query parameter `?` or a fragment `#` directly after the domain (e.g., `https://example.com?foo=1`).
**Action:** When implementing fast-path string parsers for URLs, calculate the end of the authority section by finding the minimum valid index among `/`, `?`, and `#`, and gracefully fall back to the native `new URL()` implementation (within a `try/catch` block) if the input isn't a simple string or contains complex tokens like credentials (`@`), ports (`:`), IPv6 boundaries (`[`), or encoded sequences (`%`).
## 2024-06-21 - [Array over Set for small unique collections in tight loops]
**Learning:** While `Set` objects provide `O(1)` time complexity for membership checks, the constant factor overhead of memory allocation (instantiating the `Set` object) and subsequent conversion back to an array (`Array.from()`) is significantly higher than a simple `O(N)` scan on a plain array for very small collections (like unique IP addresses or links in an email).
**Action:** When extracting a tiny number of unique items in a tight execution path, use a plain JavaScript array and enforce uniqueness by checking `array.indexOf(item) === -1` before pushing, completely avoiding `Set` allocation.
## 2024-07-28 - Fast string substring and boundary scanning vs Regex
**Learning:** Using heavy global Regex (e.g. `URGENCY_REGEX_COMBINED`) to scan for a short dictionary of keywords inside megabytes of text can cause severe blocking on the main thread. While testing for word boundaries `\b`, standard `String.prototype.indexOf` combined with a manual boundary checking via `charCodeAt` completely avoids the Regex engine instantiation and processing overhead, making it >10x faster in microbenchmarks and much more memory efficient since it prevents allocation during regex match group extraction.
**Action:** Replace known-dictionary multi-term regexes with simple `indexOf` loops inside hot text-parsing paths, especially if the text chunks are very large, while utilizing fast arithmetic boundary checks (`charCodeAt`) to match exactly the required punctuation/whitespace rules.
